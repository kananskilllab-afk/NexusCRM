API Error Handling Standard (All Pages)
•
All API calls wrapped in React Query. On error: toast notification appears in top-right corner with message from API
error.message field.
•
401 Unauthorized → automatic redirect to /login with returnUrl preserved in query param.
•
403 Forbidden → redirect to /403 page showing role explanation and contact-owner CTA.
•
429 Rate Limited → toast "Too many requests. Please wait X seconds." with countdown.
•
500 Server Error → toast "Something went wrong. Our team has been notified." + Sentry event captured.
•
Network Offline → banner appears at top of page: "No internet connection. Changes will sync when reconnected."
Toast Notification
Standard
All user feedback uses a toast system (react-hot-toast). Success = green. Error = red. Warning =
amber. Info = blue. Duration: 4s for success, 8s for errors (user needs time to read). Errors include a
"View Details" link that opens error details in a modal for debugging.

Sign In — Complete Logic Flow
• Step 1: onSubmit fires React Hook Form handleSubmit()
• Step 2: Zod schema validates { email: z.string().email(), password: z.string().min(8) }
• Step 3: If validation fails → setError() on each field → return early
• Step 4: setIsLoading(true) → disable button → show spinner
• Step 5: POST /api/v1/auth/login { email, password, rememberMe }
• Step 6: Server validates credentials via Clerk SDK / bcrypt compare
• Step 7: If MFA enabled on account → return { requiresMfa: true, tempToken: "..." } → redirect to /mfa
• Step 8: On success → server sets httpOnly cookie (voyage_session) → returns { user, tenant, role, permissions }
• Step 9: Client stores user+tenant+role in Zustand authSlice
• Step 10: Redirect to localStorage.getItem("returnUrl") || "/dashboard"
• Step 11: Clear returnUrl from localStorage after redirect
• Step 12: Log successful login to audit_events table with IP address and user agent

Drag & Drop Stage Transition — Complete Logic
• 1. User drags card (onDragStart): card gets semi-transparent overlay. Snapshot of source position saved.
• 2. User drops card (onDragEnd): react-beautiful-dnd provides { source.droppableId, destination.droppableId }.
• 3. If destination is same as source: no-op, restore card position.
• 4. Optimistic UI: immediately update local React Query cache to show card in new column.
• 5. PATCH /api/v1/bookings/:id { stageId: destination.droppableId } fired in background.
• 6. Server: validates user has permission to move this booking. Updates bookings.stage_id. Creates audit_events record.
• 7. Server: evaluates automation rules for new stage (time triggers, email sequences, score changes).
• 8. Server: broadcasts stage change event via Socket.io to all agents connected to this tenant.
• 9. On API error: revert card to source position. Show error toast. Log error to Sentry.
• 10. On API success: React Query invalidates /bookings/:id cache. Stage counts update.
